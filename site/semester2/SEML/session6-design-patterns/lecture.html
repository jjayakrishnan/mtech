Session 6 · Overview

Design Patterns for ML Production Systems

Three architectural pillars: Event-Driven Architecture decouples producers from consumers; Model Registry provides versioned governance for ML artifacts; Batch vs Online Serving is the critical decision matching prediction latency to business need.

The common thread: All three patterns address the gap between experimentation (where anything goes) and production (where reliability, traceability, and latency are non-negotiable).
Event-Driven Architecture

Direct Calls vs Event-Driven Architecture

The fundamental choice: does Service A call Service B directly, or does A emit an event and B listens? The interactive shows how this changes coupling as the system grows.

Interactive: Coupling Comparison

Direct calls: every service knows about every other service. Adding a new consumer requires modifying the producer.
The N²-to-N problem: With direct calls, N services create N(N-1)/2 potential connections. With EDA, N services connect to 1 broker = N connections. At N=10: 45 vs 10.
Pub-Sub & Kafka

Kafka: The Event Streaming Platform

Apache Kafka implements EDA at scale. Producers write events to topics; consumers subscribe to topics. Events are retained and replayable.

EDA in ML

Event-Driven ML Pipelines

In ML systems, EDA creates a flywheel: predictions logged as events trigger downstream monitors, which trigger retraining, which updates the model, which improves predictions.

Kafka in ML: Producer side

Inference service writes to predictions-topic: {user_id, features, prediction, timestamp}. No downstream service is coupled — they subscribe independently.

Kafka in ML: Consumer side

Drift Monitor subscribes, computes PSI. If drift > 0.2, writes to retrain-trigger-topic. Training Service subscribes, retrains model, uploads to MLflow registry.

Real-time feature pipeline

User events (clicks, purchases) → Kafka → Flink/Spark Streaming → Feature Store update → Model reads fresh features for next prediction. Sub-second feature freshness.

Audit and compliance

Every prediction stored in Kafka with full feature vector. Replay for audit: "What did the model predict for user X at time T and why?" Regulatorily required in finance/healthcare.
Model Registry

Why a Model Registry?

Without a registry, model files live on data scientists' laptops. Version control for code is solved (Git); model registry solves the same problem for ML artifacts.

The dependency problem: Model v3.2.1 requires Python 3.9, scikit-learn 1.0.2, feature schema v7, and was trained on data slice 2024-Q3. Without a registry, this metadata is lost. The registry is the Git + GitHub for trained models.

Serving Patterns

Batch Inference vs Online Serving

The most important ML serving decision: predict in scheduled batches, or respond to individual real-time requests? The choice depends on latency tolerance, throughput, and cost.

DimensionBatch InferenceOnline Serving
LatencyHours (scheduled job)Milliseconds (synchronous response)
CostLow (spot instances, scheduled)High (always-on GPU/CPU servers)
ThroughputMillions of records/hourHundreds of req/sec (with GPU scaling)
FreshnessStale by batch intervalReal-time (latest features)
ExampleNightly churn scores; daily recommendationsFraud detection; real-time product ranking
ToolsSpark, Airflow, BigQuery MLFastAPI + TorchServe, Vertex AI Endpoints

Always Batch

Email recommendations, churn scores, credit risk reports, fraud analytics (post-hoc), ad targeting pre-computation

Always Online

Fraud detection (blocking), real-time pricing, content ranking on scroll, chatbot/LLM responses

Hybrid

Pre-compute base scores (batch) + freshen with real-time signals (online delta). Saves cost while improving freshness.
