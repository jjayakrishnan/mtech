Session 7 · Overview

Agentic AI, SAGA & Blackboard

This final session bridges cutting-edge AI systems with classical distributed computing patterns. Agentic AI systems need the same fault-tolerance guarantees as microservices. SAGA and Blackboard are the patterns that deliver them.

The convergence: LLM-powered agents are distributed systems. They call tools (external APIs), maintain state, and must handle failures. SAGA handles failure compensation; Blackboard handles shared-state coordination. Every production agent system uses both.
Agentic AI

What is an Agent?

Three competing definitions — each emphasises a different level of autonomy. Together they describe the full spectrum.

DefinitionSourceKey Emphasis
"Perceives environment; takes actions; pursues goals"Russell & Norvig (AIMA)Environment loop; deliberative goal-pursuit
"Uses an LLM to determine workflow; calls external tools autonomously"AWS/Anthropic usageLLM as the reasoning engine; tool use; self-determined flow
"Human-in-the-loop automation that adapts to changing context"Enterprise AI practitionersPractical deployment; oversight; context-adaptation
What all three share: The agent perceives, reasons, and acts — then perceives again. It is a loop, not a single inference.

Tools & Planning

Agent Tools and Planning Strategies

Tool categories

Read tools: web_search, read_file, query_db — gather information without side effects.
Write tools: send_email, create_issue, deploy_code — take actions with consequences.

Planning strategies

ReAct: interleave Reasoning (Thought) + Action + Observation in a loop.
Tree of Thought: explore multiple plan branches, backtrack on dead ends.
Plan-and-Execute: first produce a full plan, then execute step by step.

Memory types

In-context: conversation history in the prompt window.
External: vector DB (semantic recall) + structured DB (facts).
Episodic: compressed summaries of past runs.

Human-in-the-loop

Dangerous write-tool calls pause for approval. The agent presents: "I plan to send this email. Approve?" Human approves → execution resumes. Keeps autonomy; preserves oversight.
SAGA Pattern

SAGA: Distributed Transactions Without 2PC

A SAGA is a sequence of local transactions, each with a compensating transaction that undoes its effect if a later step fails. It replaces the two-phase commit (2PC) which doesn't scale across microservices.

The travel booking analogy: Book flight → Book hotel → Book car. If car booking fails, you must cancel the hotel AND the flight. These cancellations are the compensating transactions. The SAGA guarantees they execute — even if services restart mid-way.
StepLocal TransactionCompensating Transaction
1Create order (status: PENDING)Reject order (status: REJECTED)
2Reserve inventory (qty -= N)Release inventory (qty += N)
3Charge payment (debit card)Refund payment (credit card)
4Schedule delivery (create shipment)Cancel delivery
5Approve order (status: APPROVED)N/A (final step)

Blackboard Pattern

The Blackboard Pattern

Blackboard: a shared data store (the "blackboard") that multiple specialist components (Knowledge Sources) read from and write to. A controller decides which KS runs next.

The classroom analogy: A teacher writes a partial maths problem on the board. Student A (algebra expert) solves part of it. Student B (geometry expert) builds on that. The teacher (controller) decides who goes next. The blackboard holds the evolving solution — no student talks directly to another.

Three components

Blackboard: shared working memory. All partial solutions, facts, hypotheses.
Knowledge Sources (KS): independent specialists. Each checks the board; acts if it can contribute.
Controller: decides which KS runs next. Priority queue, opportunistic, or fixed cycle.

Classic uses

HEARSAY-II speech recognition (1970s NASA). Seismic data interpretation. Multi-agent planning. Anything where: the solution space is large, problem decomposition is unknown upfront, multiple independent experts contribute incrementally.

Professor's Notes from Webinar

These highlights are drawn directly from Prof. Shreyas Suresh Rao's spoken transcripts of the Session 7 and Session 8 webinars (covering design patterns, MLflow, agentic AI, and the Saga pattern). Items marked Exam Signal are explicitly flagged for the midsem; Key Concept items are stressed definitions; Professor's Comparison items are A vs B contrasts he drew.

⚑ Exam Signal

Midsem syllabus boundary: "We have three pending items: Saga pattern orchestration/choreography mechanisms, and the Blackboard pattern. With that, we will complete the syllabus for the midterm exam." Sessions 1–7 (plus Saga) form the midsem scope. The Blackboard pattern was covered but the professor confirmed: "I have not asked any question on the Blackboard pattern — either in the regular or the makeup exam."

⚑ Exam Signal

Exam format (explicitly stated in Session 8): "You can expect: (1) Agree or disagree with justification; (2) Differentiate between X and Y; (3) Scenario-based questions — a scenario is given, you answer within that scenario. No define/explain questions. 5 or 6 questions, 30 marks total, closed book, no optional questions, about 90 minutes." No code will be asked.

⇄ Professor's Comparison

Batch vs Real-time (Online) Inference: Batch = accumulate data, run a scheduled job (e.g., every 24 hours), apply model to accumulated dataset, get predictions — "also called offline prediction." Real-time = API takes user prompt, runs model, returns response immediately — "also called on-demand inference." The professor said: "If you just remember this diagram, everything I said can be easily inferred."

★ Key Concept

When to use Batch vs Real-time: Batch: customer segmentation, ETL pipelines, nightly recommendation refresh, credit risk scoring — "not an immediate requirement." Real-time: fraud detection ("better to block the card immediately"), dynamic pricing (Ola/Uber in rain), spam filtering on incoming email, any API-based service.

⇄ Professor's Comparison

Traditional AI vs Agentic AI: Traditional (predictive/generative) = responds to prompt, single/multi-turn interaction, no tool access, stateless — "treats every input as a fresh request." Agentic AI = "pursues goals over time, multi-step reasoning, tool usage, memory and context, can flex goals with limited direct supervision."

⇄ Professor's Comparison

Chatbot vs Agent: "Plan my trip to Goa: Chatbot gives you a list of beaches, you have to prompt again. Agent searches flights, checks hotels, adds to calendar — from a single prompt. Agent takes autonomous multi-step actions." This is the key differentiator the professor used to illustrate agentic AI.

★ Key Concept

Saga pattern — purpose and compensating transactions: "Solves the distributed transaction problem. In microservices, every service has its own local database. Saga ensures that a transaction spanning multiple services either all succeeds or is all rolled back via compensating transactions." For every forward transaction there is a compensating transaction: Create order → Cancel order; Make payment → Refund payment; Update inventory → Restore inventory.

⇄ Professor's Comparison

Saga Orchestration vs Choreography: Orchestration = central orchestrator calls each service in sequence; if one fails, it calls compensating transactions in reverse. Choreography = no central component; each service publishes an event after completing its task; "choreography is always event-driven." General guidance: "For N microservices, people prefer orchestration. If having 10–15 services, keeping track of events and publishers/consumers becomes very challenging."

★ Key Concept

Saga + CQRS combination: "CQRS + Saga is a very commonly recommended combination. AWS explicitly recommends the Saga pattern for agentic AI workflows." The Saga pattern also applies in agentic AI — 'services' become 'agents', transactions become 'agent tasks', and compensating transactions become agent rollback actions.
