Session 7 · Overview

Agentic AI, SAGA & Blackboard

This final session bridges cutting-edge AI systems with classical distributed computing patterns. Agentic AI systems need the same fault-tolerance guarantees as microservices. SAGA and Blackboard are the patterns that deliver them.

The convergence: LLM-powered agents are distributed systems. They call tools (external APIs), maintain state, and must handle failures. SAGA handles failure compensation; Blackboard handles shared-state coordination. Every production agent system uses both.
Agentic AI

What is an Agent?

Three competing definitions — each emphasises a different level of autonomy. Together they describe the full spectrum.

DefinitionSourceKey Emphasis
"Perceives environment; takes actions; pursues goals"Russell & Norvig (AIMA)Environment loop; deliberative goal-pursuit
"Uses an LLM to determine workflow; calls external tools autonomously"AWS/Anthropic usageLLM as the reasoning engine; tool use; self-determined flow
"Human-in-the-loop automation that adapts to changing context"Enterprise AI practitionersPractical deployment; oversight; context-adaptation
What all three share: The agent perceives, reasons, and acts — then perceives again. It is a loop, not a single inference.

Tools & Planning

Agent Tools and Planning Strategies

Tool categories

Read tools: web_search, read_file, query_db — gather information without side effects.
Write tools: send_email, create_issue, deploy_code — take actions with consequences.

Planning strategies

ReAct: interleave Reasoning (Thought) + Action + Observation in a loop.
Tree of Thought: explore multiple plan branches, backtrack on dead ends.
Plan-and-Execute: first produce a full plan, then execute step by step.

Memory types

In-context: conversation history in the prompt window.
External: vector DB (semantic recall) + structured DB (facts).
Episodic: compressed summaries of past runs.

Human-in-the-loop

Dangerous write-tool calls pause for approval. The agent presents: "I plan to send this email. Approve?" Human approves → execution resumes. Keeps autonomy; preserves oversight.
SAGA Pattern

SAGA: Distributed Transactions Without 2PC

A SAGA is a sequence of local transactions, each with a compensating transaction that undoes its effect if a later step fails. It replaces the two-phase commit (2PC) which doesn't scale across microservices.

The travel booking analogy: Book flight → Book hotel → Book car. If car booking fails, you must cancel the hotel AND the flight. These cancellations are the compensating transactions. The SAGA guarantees they execute — even if services restart mid-way.
StepLocal TransactionCompensating Transaction
1Create order (status: PENDING)Reject order (status: REJECTED)
2Reserve inventory (qty -= N)Release inventory (qty += N)
3Charge payment (debit card)Refund payment (credit card)
4Schedule delivery (create shipment)Cancel delivery
5Approve order (status: APPROVED)N/A (final step)

Blackboard Pattern

The Blackboard Pattern

Blackboard: a shared data store (the "blackboard") that multiple specialist components (Knowledge Sources) read from and write to. A controller decides which KS runs next.

The classroom analogy: A teacher writes a partial maths problem on the board. Student A (algebra expert) solves part of it. Student B (geometry expert) builds on that. The teacher (controller) decides who goes next. The blackboard holds the evolving solution — no student talks directly to another.

Three components

Blackboard: shared working memory. All partial solutions, facts, hypotheses.
Knowledge Sources (KS): independent specialists. Each checks the board; acts if it can contribute.
Controller: decides which KS runs next. Priority queue, opportunistic, or fixed cycle.

Classic uses

HEARSAY-II speech recognition (1970s NASA). Seismic data interpretation. Multi-agent planning. Anything where: the solution space is large, problem decomposition is unknown upfront, multiple independent experts contribute incrementally.
