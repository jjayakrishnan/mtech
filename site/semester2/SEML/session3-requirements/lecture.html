Session 3 · Overview

Requirements Engineering for ML Systems

Before writing a single line of ML code, you must answer three questions: Should I use ML at all? What goals must the system achieve? How will I know if it is working? This session provides the tools to answer all three.

Running example: Every concept in this session is illustrated with a credit risk prediction system at a bank — a realistic, complete ML use case.
Geoff Hulten

When Should You Use ML?

ML is not always the right tool. Hulten's framework identifies three situations where ML genuinely outperforms rule-based approaches.

Interactive: ML Decision Space

Case 1: Intrinsically hard — language, ambiguity, context. Rules cannot capture human nuance.

Case 1: Intrinsically Hard

Human language is ambiguous, sarcastic, contextual, culturally loaded. No finite set of rules captures this. ML learns patterns from millions of examples.

Case 2: Big Scale

Recommending one song from 100 million tracks to 500 million users. Manual rules are unmanageable. ML learns a personalised embedding space for every user.

Case 3: Changing Patterns

Fraud evolves. Fraudsters adapt the moment a rule blocks them. An ML model retrained weekly continuously adapts — a fixed rule set does not.
When NOT to use ML: If a business rule can be written clearly ("if age < 18, block purchase"), write the rule. ML adds cost and uncertainty you don't need.
Requirements Engineering

The Probabilistic Shift

Traditional requirements engineering assumes deterministic systems. ML is probabilistic — the same input may produce different outputs depending on model version and training data.

DimensionTraditional REML RE
SpecificationClear, exhaustive rulesMeasurable performance targets on data
CorrectnessBinary (passes/fails)Probabilistic (correct 93% of the time)
Data dependencyNoneAll predictions depend on training data quality
Change triggerBusiness rule changesData distribution changes (drift)
TestingUnit tests with known outputsStatistical evaluation on held-out datasets

Trade-offs

Goals Conflict: Trade-offs Are Inevitable

Goals at different levels often pull in opposite directions. A software architect must make explicit trade-offs.

Accuracy vs Latency

A larger model is more accurate but slower. The credit risk system needs AUC ≥ 0.85 AND response in under 2 seconds. Choose a model that satisfies both constraints.

Accuracy vs Interpretability

XGBoost achieves AUC 0.88 (excellent) but is a black box. Logistic regression achieves AUC 0.81 but explains its decisions. Regulations may require explanation — forcing the trade-off.

Cost vs Quality

GPT-4 gives the best answers but costs $0.03 per call. At 1M calls/day, that's $30,000/day. A fine-tuned smaller model costs $0.001/call but may miss edge cases.

Fairness vs Accuracy

Optimising overall accuracy may harm minority groups with less training data. Fairness constraints (demographic parity, equal opportunity) may slightly reduce overall accuracy.
GR4ML Introduction

From Goals to Requirements: The GR4ML Framework

GR4ML (Goal and Requirement for ML) is a formal notation that traces the chain from high-level business goals down to specific ML data and algorithm choices. It has three views.

The chain: Business View (WHY?) → Analytics Design View (WHAT to predict?) → Data Preparation View (HOW to prepare the data?). All three must be specified before development begins.
Organisation asks: "Should we approve this loan?"
→ GR4ML Business View: this is a DecisionGoal by an Actor (Case Worker) serving a StrategicGoal (good lending)
→ GR4ML Analytics View: the decision requires a PredictionGoal: "What is the probability of default?"
→ GR4ML Data View: the prediction requires DataPreparationTasks on CreditBureau + TransactionHistory data

GR4ML: View 3

The Data Preparation View

The Data Preparation View maps raw data sources to the clean, feature-engineered dataset that the ML model trains on. Every preparation task must be specified explicitly.

Entity (Data Sources)

LoanApplication (50 columns), CreditBureau (30 columns), TransactionHistory (3 years). These are the raw inputs before any preparation.

Data Cleaning

Remove 3,200 rows with missing income (3.2%). Impute 800 rows with missing credit score using median by age bracket. Operator: Python/pandas.

Data Reduction

Drop 22 columns with >40% missing values. Retain 35 predictive features identified by domain experts and feature importance analysis.

Normalisation & Encoding

Min-max scale: loan_amount, annual_income, total_debt (→ [0,1]). One-hot encode: loan_purpose (12 categories → 12 binary columns).
Output: 97,000 clean rows, 58 features, ready for XGBoost training. This data pipeline must be versioned and reproducible — the same pipeline must produce the same dataset from the same raw data.
Measurement

From Abstract Goals to Concrete Measures

A goal like "improve chatbot usefulness" is unmeasurable. You must define measures — concrete, quantifiable indicators — before you can track progress.

Measure vs Metric: A measure is the method ("task success rate"). A metric is the measured value ("83% this week"). A good measure directly relates to a goal, is quantifiable, and is practical to collect.

Measurement Quality

Accuracy vs Precision: The Darts Board

These two properties describe how good your measurement system is — and they are independent. You need both to trust your metrics. Use the sliders to see the darts analogy.

Interactive: Darts Board

Adjust sliders to see how accuracy and precision affect dart groupings.
In ML:
Accuracy of measurement ≈ low bias (predictions centred on true values)
Precision of measurement ≈ low variance (predictions consistent across runs)
Bias–variance trade-off is the ML counterpart of accuracy–precision
Wrap-up

Summary: Session 3 Key Points

When to Use ML

Hulten's 3 cases: intrinsically hard, big scale, or changing patterns. If none apply, write a rule instead.

Goal Alignment

Four levels: Model → System → User → Organisational. All must align. Conflicts require explicit trade-offs.

GR4ML

Three views: Business (WHY) → Analytics Design (WHAT) → Data Preparation (HOW). Trace from strategy to algorithm to data pipeline.

Measurement

Vague goals must become concrete measures before development. Both accuracy (unbiased) and precision (consistent) of measurement matter.
