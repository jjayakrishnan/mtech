Alpha-Beta Pruning & Monte Carlo Tree Search

Lesson 8 of 8 • Static Evaluation • Alpha-Beta Pruning • Move Ordering • MCTS (4 Phases) • Stochastic Games

1. Static Evaluation Functions

Minimax (CS7) assumes we can reach terminal states. In real games like chess, the tree is too deep — we cut off search at depth \(d\) and apply a static evaluation function \(\text{Eval}(s)\) to estimate the state's value without searching further.

What makes a good evaluation function?

  • Must agree with the terminal utility function on terminal states.
  • Should be fast to compute.
  • Should be strongly correlated with the actual chance of winning.
  • Combines material features (piece counts/values) and positional features (piece placement, control, pawn structure).

Chess example: weighted linear evaluation

$$\text{Eval}(s) = w_1 f_1(s) + w_2 f_2(s) + \cdots + w_n f_n(s)$$

Where typical features \(f_i\) and weights \(w_i\) are:

FeatureWeightNote
Pawn advantage1+1 per extra pawn
Knight / Bishop advantage3+3 per extra minor piece
Rook advantage5+5 per extra rook
Queen advantage9+9 per extra queen
Mobility0.1+0.1 per extra legal move available
H-minimax: Replace terminal states with cutoff states evaluated by \(\text{Eval}(s)\). The agent searches to depth \(d\), then evaluates. Deeper search + better evaluation = stronger play.
Static Evaluation Function — General Formula
$$\text{SEV} = \sum(\text{Utility of MAX pieces}) - \sum(\text{Utility of MIN pieces})$$

Properties of a good SEV

  • Higher values = better for MAX; lower values = better for MIN. Zero means an even position.
  • Fast to compute — evaluated at thousands of nodes per second; it cannot be expensive.
  • Correlates with actual winning chance — a better SEV score should genuinely mean you're more likely to win.

Chess piece values (standard)

PieceValue
Queen9
Knight / Horse3
Bishop3
Pawn / Soldier1

Worked Examples — Computing SEV

Example 1: White (MAX) has Horse + Bishop; Black (MIN) has Horse + Soldier + Queen.

  • MAX pieces: \( 3 + 3 = 6 \)
  • MIN pieces: \( 3 + 1 + 9 = 13 \)
  • SEV = \( 6 - 13 = -7 \) → Black has the advantage (negative = MIN winning).

Example 2: White (MAX) has Horse + Bishop + Soldier; Black (MIN) has Bishop + Soldier.

  • MAX pieces: \( 3 + 3 + 1 = 7 \)
  • MIN pieces: \( 3 + 1 = 4 \)
  • SEV = \( 7 - 4 = +3 \) → White has the advantage (positive = MAX winning).

Tic-Tac-Toe SEV method

There are no piece values in Tic-Tac-Toe. Instead, use a winning-line count approach:

  1. Fill all empty cells with X → count the number of lines that X could complete (rows, columns, diagonals). Call this \( W_X \).
  2. Fill all empty cells with O → count the number of lines that O could complete. Call this \( W_O \).
  3. \( \text{SEV} = W_X - W_O \)

A higher value means X (MAX) has more winning opportunities.

📋 Exam Scope — Static Evaluation

"Static values will either be given directly in the tree, OR a formula like chess piece values will be given. You will not be asked to derive chess piece values from scratch."

2. Alpha-Beta Pruning

Alpha-beta pruning is an enhancement to Minimax that skips branches that cannot affect the final decision. It produces the same result as full Minimax but explores far fewer nodes.

The two values

  • \(\alpha\) (alpha): The best value MAX has found so far along the current path. MAX wants to maximise, so \(\alpha\) is the lower bound on MAX's score. Starts at \(-\infty\).
  • \(\beta\) (beta): The best value MIN has found so far along the current path. MIN wants to minimise, so \(\beta\) is the upper bound on MIN's score. Starts at \(+\infty\).
Prune condition
$$\text{Prune remaining children when } \beta \leq \alpha$$

MIN will never allow this path if MAX already has something better; MAX will never use this path if MIN already has something worse.

The core idea (prof's words)

"Alpha-beta pruning gives THE SAME ANSWER as minimax — but skips branches that cannot affect the result. It's minimax made smarter."

Alpha and Beta — definitions

TermRepresentsInitialized toNode type
Alpha (\(\alpha\))Best value MAX has found so far\(-\infty\)MAX nodes
Beta (\(\beta\))Best value MIN has found so far\(+\infty\)MIN nodes
When to prune — the key condition

Prune when: \(\alpha \geq \beta\)

  • At a MIN node: if the current MIN value ≤ the parent MAX's \(\alpha\) → prune remaining siblings.
  • At a MAX node: if the current MAX value ≥ the parent MIN's \(\beta\) → prune remaining siblings.

Prof's intuition

"This MIN node is trying to find something less than 6. This MAX node is trying to find something greater than 8. Even if it finds something greater than 8, the MIN parent won't pick it — the MIN parent will choose something less than 6. So don't bother searching. That's pruning."

Traversal rules

  • Traversal: depth-first, in-order, leftmost first.
  • The leftmost child can NEVER be pruned — there is no prior sibling value to compare against.
  • Only right siblings can be pruned.
  • Root can be MAX or MIN — the problem will state which.

Alpha-Beta Algorithm Steps (from CS8 slides)

  1. At the root, initialise: \(\alpha = -\infty\), \(\beta = +\infty\).
  2. Traverse the tree to the specified depth / terminal nodes, getting the static evaluation value VAL.
  3. For every node being analysed, loop through all children until all are analysed OR \(\beta \leq \alpha\):
    1. If the current player is MAX:
      • If VAL > \(\alpha\): update \(\alpha = \text{VAL}\).
      • If \(\beta \leq \alpha\): tag this path as unpromising and prune the remaining siblings.
    2. If the current player is MIN:
      • If VAL < \(\beta\): update \(\beta = \text{VAL}\).
      • If \(\beta \leq \alpha\): tag this path as unpromising and prune the remaining siblings.

Tree structure (CS8 Example 1)

3-level tree. MAX at root (A), MIN at level 1 (B, C), MAX at leaves. Leaf values (left to right): 6, 5, 8, 10, 2, 1, 9, 12 

                A (MAX)
              /       \
           B (MIN)   C (MIN)
          /    \     /    \
        [C]   [D]  [E]   [F]
        6 5   8 10  2 1   9 12

Trace: Alpha-Beta on Example 1

  1. Initialise root A: \(\alpha = -\infty\), \(\beta = +\infty\).
  2. Go to B (MIN): \(\alpha = -\infty\), \(\beta = +\infty\) passed down.
    • Visit leaf 6 → VAL = 6. B is MIN: \(\beta = \min(+\infty, 6) = 6\).
    • Visit leaf 5 → VAL = 5. B is MIN: \(\beta = \min(6, 5) = 5\). B returns 5.
    • Back at A (MAX): \(\alpha = \max(-\infty, 5) = 5\).
  3. Go to D under B (MAX node at depth 2 in CS8's 3-layer example):
    • Visit leaf 8 → VAL = 8. D is MAX: \(\alpha = \max(5, 8) = 8\) for D's subtree.
    • Check prune: D's parent B has \(\beta = 5\). Since VAL = 8 > \(\beta = 5\) (MIN parent won't allow), prune remaining sibling (leaf 10). Node 10 is pruned.
    • B returns 5 (its \(\beta\)).
  4. Back at A: \(\alpha = 5\) (best MAX has found so far, via B).
  5. Go to C (MIN): \(\alpha = 5\) passed down.
    • Visit leaf 2 → VAL = 2. C is MIN: \(\beta = \min(+\infty, 2) = 2\).
    • Check prune: \(\beta = 2 \leq \alpha = 5\). Prune subtree F (leaves 9 and 12). They are pruned.
    • C returns 2.
  6. A (MAX): compare children B=5 and C=2. MAX picks 5. Root value = 5. Wait — revisit: leaf 1 under E: MIN sees 2 then 1, \(\beta=1\), prune F. A picks max(5, 1) = 5. ✓

Pruned nodes: leaf 10 (under D) and leaves 9, 12 (under F). Only 6 of 8 leaves evaluated.

Try It: Deeper Tree (CS8 Example 2)

4-level tree: MAX at root, MIN at level 1, MAX at level 2, MIN at leaves. Leaf values (left to right): 3, 4, 2, 1, 7, 8, 9, 10, 2, 11, 1, 12, 14, 9, 13, 16.

Apply alpha-beta. What is the root value? Which leaves get pruned?

Working through the tree (left subtree first):

  • Left MAX node (level 2, left): children are MIN nodes with leaves {3,4} and {2,1}. Left MIN = min(3,4)=3. Right MIN: leaf 2 → \(\beta=2\), check with left MIN result 3 → \(\beta=2 \leq \alpha=3\), prune leaf 1. Right MIN=2. Left MAX = max(3,2)=3.
  • Root is MAX, so \(\alpha=3\).
  • Right MAX node (level 2, right for left MIN): children {7,8} and {9,10}. Left MIN=7. Right MIN=9, then 10→still 9. Right MAX=max(7,9)=9. Left root MIN = min(3,9)=3. (\(\alpha=3\) passes to right side...)
  • Continue: root MIN gets 3 from left MAX subtree. Right MAX at level 2 eventually yields 9; but root MIN already has \(\beta=3\) → prune once right MAX exceeds 3.
  • Root value = 3. Multiple prunings occur, particularly in right subtrees once \(\alpha/\beta\) bounds tighten.
Complexity improvement from alpha-beta
$$\text{Minimax: } O(b^m) \quad \to \quad \text{Alpha-Beta: } O(b^{m/2})$$

Alpha-beta effectively halves the search depth exponent — it can look twice as deep as minimax in the same time.

Quick Check: What if L3-4's left leaf were 6 instead of −10?

Would L3-4 still be pruned? Why or why not?

Yes, still pruned. Pruning of L3-4 happened because L2-B already found a value (8) ≥ L1-LEFT's \(\beta=7\). At that point, whatever L3-4 returns, L2-B ≥ 8 > 7 means L1-LEFT (MIN) would never choose L2-B anyway. The individual leaf values of L3-4 are irrelevant — the prune condition depends on L2-B's running maximum vs. the parent MIN's \(\beta\).

5. Constructing a Tree With No Pruning

A past ACI mid-semester exam question asked: "Construct leaf values for a game tree such that no branch is pruned by alpha-beta." To answer this, you need to understand exactly when pruning fires and engineer values that never trigger it.

When Does Pruning Fire?

Pruning fires at a node when \(\beta \leq \alpha\) — meaning:

  • At a MAX node: MAX has found something ≥ the MIN ancestor's best — MIN would never allow this branch. Remaining children skipped.
  • At a MIN node: MIN has found something ≤ the MAX ancestor's best — MAX would never choose this branch. Remaining children skipped.

To prevent pruning: the right child must never trigger the prune condition relative to the bounds inherited from ancestors.

The No-Pruning Rule

  • At every MAX node: values returned by children must be non-decreasing left-to-right (each right child ≥ left). Then α is never tight enough to cause a β-cutoff at a MIN ancestor.
  • At every MIN node: values returned by children must be non-increasing left-to-right (each right child ≤ left). Then β is never tight enough to cause an α-cutoff at a MAX ancestor.

Equivalently: arrange leaves so that the best move (from the current player's view) is always visited last. This is worst-case ordering — alpha-beta degenerates to full Minimax, examining every node.

Worked Example: 3-Level Tree, No Pruning

Tree: MAX root → two MIN children → two leaves each (4 leaves total).

Target: Choose leaf values so all 4 are evaluated.

        MAX (root)
        /       \
     MIN-L      MIN-R
     /   \      /   \
    5     9    7     9
  1. Root starts: \(\alpha=-\infty\), \(\beta=+\infty\).
  2. MIN-L: visit leaf 5 → \(\beta_{L}=5\). Visit leaf 9 → \(\beta_{L}=\min(5,9)=5\) (no change). MIN-L returns 5.
  3. Root MAX: \(\alpha=\max(-\infty,5)=5\).
  4. MIN-R: visit leaf 7 → \(\beta_{R}=7\). Check: \(\beta_{R}=7 \leq \alpha_{root}=5\)? No (7 > 5). So visit right leaf 9 → \(\beta_{R}=\min(7,9)=7\). MIN-R returns 7.
  5. Root MAX = max(5, 7) = 7. All 4 leaves evaluated. ✓

Why no pruning? MIN-L returned 5 (low), so \(\alpha_{root}=5\) is low. MIN-R's first child (7) sets \(\beta_R=7 > \alpha=5\), so the prune condition \(\beta \leq \alpha\) never fires.

Contrast: Same Tree, Pruning DOES Fire

Swap leaf order to: 9, 5, 7, 9

        MAX (root)
        /       \
     MIN-L      MIN-R
     /   \      /   \
    9     5    7     9
  1. MIN-L: visit leaf 9 → \(\beta_L=9\). Visit leaf 5 → \(\beta_L=\min(9,5)=5\). Returns 5.
  2. Root: \(\alpha=5\). (Same as before — MIN-L result unchanged.)
  3. MIN-R: visit leaf 7 → \(\beta_R=7 > \alpha=5\), continue. Visit leaf 9 → \(\beta_R=\min(7,9)=7\). Returns 7.
  4. Root = 7. Still no pruning here because MIN-R's first child (7) is still > α=5.

Let's try leaves 9, 5, 3, 9 to force pruning:

        MAX (root)
        /       \
     MIN-L      MIN-R
     /   \      /   \
    9     5    3     9
  1. MIN-L returns 5. Root \(\alpha=5\).
  2. MIN-R: visit leaf 3 → \(\beta_R=3\). Check: \(\beta_R=3 \leq \alpha_{root}=5\). PRUNE right leaf (9). MIN-R returns 3.
  3. Root = max(5, 3) = 5. Only 3 of 4 leaves evaluated.

Pruning fired because MIN-R found a value (3) already < α (5), so MAX would never pick this branch — no point continuing.

6. Efficiency & Move Ordering

Complexity comparison
AlgorithmTime complexity
Minimax\(O(b^m)\)
Alpha-Beta (worst case, random ordering)\(O(b^m)\)
Alpha-Beta (best case, perfect ordering)\(O(b^{m/2})\)

With perfect move ordering, alpha-beta can search twice as deep as Minimax for the same computation.

Move Ordering

Alpha-beta is most effective when good moves are explored first — they set tight bounds early, enabling more pruning later. This is move ordering.

  • Without move ordering: weak bounds, few prunings, approaches \(O(b^m)\).
  • With move ordering: tight bounds early, many prunings, approaches \(O(b^{m/2})\).
  • Practical heuristic: try moves with higher static evaluation scores first.

7. Monte Carlo Tree Search (MCTS)

Alpha-beta with static evaluation works well for chess, but fails for games like Go where:

  • The branching factor is ~200 (too large for deep alpha-beta).
  • Good static evaluation functions are extremely hard to design.

MCTS avoids the need for a hand-crafted evaluation function. Instead, it estimates state values by simulating random games from that state and recording win rates.

Why MCTS? (prof)

"Go has ~400 legal moves per state (vs ~35 for chess). A full game-tree search is computationally impossible. MCTS doesn't build the full tree upfront — it runs simulations to find out which parts are worth exploring."

The balance MCTS must strike:

  • Exploitation: focus on already-promising moves (capitalize on known strengths).
  • Exploration: occasionally try uncertain moves (sometimes yields something better).

UCB1 formula (used during Selection) balances these two automatically.

MCTS Core Idea

Use results of simulations to guide growth of the game tree. Balance two goals:

  • Exploitation: focus on moves that have won often in the past.
  • Exploration: try moves that haven't been tried much (uncertainty might hide a good move).
UCB Selection Formula (Upper Confidence Bound)
$$\text{UCB}(i) = v_i + C \times \sqrt{\frac{\ln N}{n_i}}$$

\(v_i\) = win rate estimate of node \(i\) (exploitation) • \(C\) = exploration constant (tunable) • \(N\) = parent visit count • \(n_i\) = visit count of node \(i\) (more visits → explore less)

UCB Intuition

  • If \(n_i\) is small (rarely visited), the right term is large → algorithm is curious about this node.
  • If \(v_i\) is large (good win rate), the left term is large → algorithm keeps exploiting this node.
  • As \(n_i \to \infty\), the exploration bonus shrinks → converges to exploiting the best-known move.

8. MCTS: The Four Phases

Each iteration of MCTS repeats these four phases:

1. Selection → 2. Expansion → 3. Simulation → 4. Backpropagation

1. Selection Choose the most promising node

Starting from the root, repeatedly choose the child with the highest UCB score until reaching a node that has unexplored children (a leaf in the current tree, not the game).

Example: In Tic-Tac-Toe, from the current board, the algorithm chooses the move with a good past win rate but may still explore less-tried moves.

2. Expansion Add a new child node

Once selection reaches a node where not all moves have been tried, pick one unexplored move and add it as a new child node. The tree grows incrementally — one node per iteration — instead of all at once.

Example: If the selected Tic-Tac-Toe board still has 3 legal moves (top-right, middle-left, bottom-center), expand by adding top-right as a new node.

3. Simulation (Rollout) Play to the end randomly

From the newly expanded node, play the game to a terminal state using a simulation policy (often random, or lightweight heuristic). Record the outcome (win/loss/draw).

The purpose is not to play perfectly — just to estimate whether the new move leads to a win, loss, or draw.

4. Backpropagation Update all ancestors

Propagate the simulation result back up through all nodes visited during selection. Each node on the path updates:

  • Visit count: \(n_i \mathrel{+}= 1\)
  • Win score: \(w_i \mathrel{+}= 1\) if the simulation resulted in a win (for that player)
  • Win rate: \(v_i = w_i / n_i\)

MCTS Pseudocode (T1)

function MONTE-CARLO-TREE-SEARCH(state) returns action:
  tree ← NODE(state)
  while IS-TIME-REMAINING() do
    leaf  ← SELECT(tree)          // phase 1
    child ← EXPAND(leaf)          // phase 2
    result ← SIMULATE(child)      // phase 3
    BACK-PROPAGATE(result, child)  // phase 4
  return move in ACTIONS(state) whose node has
         highest number of playouts
Algorithm Summary
while time_remaining:
  1. SELECT:  traverse via UCB1 to find a node with unexplored children
  2. EXPAND:  add one new child (initialize to 0/0)
  3. SIMULATE: play randomly to terminal state
  4. BACKPROPAGATE: update wins/plays up the path
Return: root child with the highest PLAY COUNT (most explored)

📋 Exam Scope — MCTS

"As long as you know what the 4 steps are and what happens in each — that's enough. You won't get a full numerical MCTS problem like alpha-beta. The idea is: MCTS uses smart simulations to identify which nodes to explore further."

Node storage

Each tree node stores: wins / total_plays (e.g., 2/3 = won 2 of 3 simulations). The loop repeats until the time budget runs out.

Back propagation — common exam mistake

"If the simulation resulted in Black winning, and you're updating from White's perspective: White's WIN count stays the same — only the PLAY count increments. Only the actual winner gets the win credit. Students often forget this."

9. MCTS Worked Example

Starting state (from CS8 slides)

Root node has been visited 7 times with 4 wins: 4/7. It has three children:

Childwins/visitsWin rate \(v_i\)
Child L (left)2/30.667
Child M (middle)1/30.333
Child R (right)0/10.000

One MCTS iteration (C = 1.41 ≈ \(\sqrt{2}\))

  1. Selection Compute UCB for each child (\(N = 7\)):
    • Child L: \(0.667 + 1.41\sqrt{\ln 7 / 3} = 0.667 + 1.41 \times 0.856 = 1.875\)
    • Child M: \(0.333 + 1.41\sqrt{\ln 7 / 3} = 0.333 + 1.209 = 1.542\)
    • Child R: \(0.000 + 1.41\sqrt{\ln 7 / 1} = 0.000 + 2.635 = 2.635\)
    Select Child R (highest UCB = 2.635) — it's under-explored.
  2. Expansion Child R has 1 visit but no children in the tree yet. Expand by adding a new child node (0/0).
  3. Simulation From the new node, play randomly to the end. Suppose the simulation result: win (1).
  4. Backpropagation Update the path: new node → Child R → Root.
    • New node: 1/1
    • Child R: wins \(0 \to 1\), visits \(1 \to 2\) → 1/2
    • Root: wins \(4 \to 5\), visits \(7 \to 8\) → 5/8

Try It: UCB Selection

Root has been visited \(N = 10\) times. Three children with stats: A = 3/5 (wins/visits), B = 1/4, C = 0/1. Using \(C = \sqrt{2} \approx 1.414\), which child does MCTS select next?

Compute UCB (\(\ln 10 \approx 2.303\)):

  • A: \(3/5 + 1.414\sqrt{2.303/5} = 0.6 + 1.414 \times 0.679 = 0.6 + 0.960 = \mathbf{1.560}\)
  • B: \(1/4 + 1.414\sqrt{2.303/4} = 0.25 + 1.414 \times 0.758 = 0.25 + 1.072 = \mathbf{1.322}\)
  • C: \(0/1 + 1.414\sqrt{2.303/1} = 0 + 1.414 \times 1.518 = \mathbf{2.146}\)

Select C (UCB = 2.146) — it has only been visited once, so the exploration bonus dominates.

10. Stochastic Games

Real games often include chance (dice, card shuffles). These are stochastic games. Minimax is extended with chance nodes.

Expectiminimax

Add chance nodes between MAX and MIN nodes. A chance node computes the expected value over all outcomes:

$$\text{Expectiminimax}(s) = \sum_{r} P(r) \cdot \text{Minimax}(\text{RESULT}(s, r))$$

where \(r\) ranges over possible random outcomes and \(P(r)\) is the probability of each.

Examples of stochastic games

GameChance elementType
BackgammonDice rolls2-player, perfect info + chance
PokerCard dealing2-player, imperfect info + chance
MonopolyDice + cardsMulti-player + chance
Pruning in Expectiminimax: Alpha-beta pruning still works at MAX/MIN nodes. At chance nodes, pruning is harder — you need bounds on the probability distribution to prune.

Alpha-Beta Pruning

  • \(\alpha\): best value MAX can guarantee (lower bound). Starts at \(-\infty\).
  • \(\beta\): best value MIN can guarantee (upper bound). Starts at \(+\infty\).
  • Prune when: \(\beta \leq \alpha\).
  • MAX node: update \(\alpha = \max(\alpha, \text{VAL})\); prune if \(\beta \leq \alpha\).
  • MIN node: update \(\beta = \min(\beta, \text{VAL})\); prune if \(\beta \leq \alpha\).

MCTS Four Phases

PhaseWhat it does
SelectionFollow UCB scores from root to a leaf with unexplored children
ExpansionAdd one new child node for an unexplored move
Simulation (Rollout)Play randomly to a terminal state; record win/loss
BackpropagationUpdate visit count + win count up the path to root

UCB Formula

  • \(\text{UCB}(i) = v_i + C \sqrt{\ln N / n_i}\)
  • Large \(n_i\) (many visits) → small exploration bonus → exploit
  • Small \(n_i\) (few visits) → large exploration bonus → explore
