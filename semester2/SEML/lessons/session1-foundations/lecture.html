Session 1 · Introduction

What this session is about

Session 1 builds the vocabulary for the entire course. You will understand what Software Engineering, Data Science, and Machine Learning are, how they relate, and why building ML systems requires different thinking than building traditional software.

Analogy: Traditional software is a recipe: fixed steps, deterministic output. ML software is a chef who learns from millions of meals and improvises. The engineering challenge is completely different — and this session explains why.
By the end of this session you will know: the 6 SDLC phases, the 3 generations of ML, the 5 major ML domains, what a Foundation Model is, and the key differences between SE and ML engineering.
Software Engineering

The Software Development Life Cycle

Every software product follows a structured process from idea to shipped product. This is the SDLC — the backbone of software engineering.

Analogy: Building a house. First the owner explains needs (requirements). Then blueprints (design). Then construction (implementation). Then inspection (testing). Then move in (deployment). Then plumbing fixes (maintenance). Software teams do the same six steps.

Software Engineering

Evolution to Cloud-Native

Software went from shipping once a year (traditional SDLC) to shipping multiple times per day (cloud-native). Three innovations made this possible.

Agile

Work in 2-week sprints. Ship working software continuously. Welcome requirement changes. Focus on collaboration.

DevOps

Dev and Ops teams share responsibility. Automate testing, integration, and deployment (CI/CD). "You build it, you run it."

Microservices

Break the monolith into small independent services. Each service owns its data and deploys independently.

Cloud + Containers

Docker for packaging; Kubernetes for orchestration; AWS/Azure/GCP for elastic compute. Scale up in seconds.
Key insight: Every ML team operates within some variant of these practices. ML adds new steps (data preparation, model training, model monitoring) into the CI/CD pipeline.
Data Science

What is Data Science?

Data Science is the discipline of extracting actionable insights from data — through collection, cleaning, analysis, modelling, and visualisation.

Analogy: A detective who analyses clues (data) to reach a conclusion (insight). The detective doesn't know the answer in advance — they let the evidence guide them. Data scientists let the data speak.

The 5-step data science workflow:

1. Collect

Gather raw data from databases, APIs, sensors, surveys, web scraping.

2. Preprocess

Fix missing values, remove duplicates, normalise formats, handle outliers.

3. Analyse (EDA)

Visualise distributions, find correlations, spot anomalies, understand the data.

4. Model

Train a statistical or ML model that captures patterns for prediction.

5. Communicate

Share findings through dashboards, reports, or APIs that decision-makers can act on.
AI / ML / Data Science

The Convergence: AI, ML, Deep Learning, Data Science

These four terms describe overlapping disciplines. Understanding the precise hierarchy prevents confusion.

Interactive: Nested Circles

Click a ring to see what it covers.

  • Machine Learning (ML) — a subset of AI; machines learn from data instead of being programmed with rules.
  • Deep Learning — a subset of ML using multi-layer neural networks; excels on images, text, audio.
  • Data Science — overlaps all three; focuses on extracting insights and communicating them.
Machine Learning

ML Basics: The 3-Variable Complexity

Machine Learning is a branch of AI where a program learns patterns from data without being hand-programmed with rules. But ML systems are uniquely complex because three things — not one — determine their behaviour.

Analogy: Traditional software = one moving part: the code. ML = three moving parts: Data, Model, and Code. Change any one and the output changes. That makes debugging three times harder.

Interactive: 3-Variable Complexity

ML: 3 variables affect output. Traditional: 1 variable (code) affects output.
Watch out: The most dangerous ML bug is a silent one. A software bug crashes and announces itself. A data bug makes quietly wrong predictions for weeks. You must test Data, Model, and Code together.

ML Pipeline

Static Pipeline and Production Pipeline

The full ML pipeline has two linked loops: a training (static) loop that builds the model, and a production loop that monitors and retrains it.

Static pipeline: Collect Data → Feature Engineering → Train Model → Evaluate → (if not good enough) loop back to Collect.
Production pipeline: Deploy Model → Monitor → (if performance drops) Retrain → redeploy. This loop runs continuously in production.
Key insight: Most lectures focus on the training pipeline. But in industry, the production pipeline is where most engineering effort goes — monitoring, drift detection, retraining triggers.
ML Domains

Types of ML Domains

ML is applied across many domains. The three most prevalent in industry are NLP, Computer Vision, and Speech Recognition. Click a domain to explore it.

Interactive: Domain Explorer

Click a domain tile to see examples and applications.

NLP — Natural Language Processing

Teaches machines to understand and generate human language. Examples: sentiment analysis, translation (Google Translate), summarisation, chatbots, question answering.

Computer Vision

Interprets visual data. Examples: object detection in autonomous vehicles, face recognition, tumour detection in X-rays, quality control on assembly lines.

Speech Recognition (ASR)

Converts spoken audio to text. Examples: virtual assistants (Alexa, Siri, Google Assistant), meeting transcription, call-centre automation.
Foundation Models

Foundation Models and Large Language Models

A Foundation Model is trained on massive, broad data and then adapted to specific tasks. LLMs are the dominant type today — trained on text, they can write, summarise, translate, reason, and generate code.

Analogy: A Foundation Model is a university graduate who spent years learning broadly (pre-training). You give them a 2-week onboarding (fine-tuning) and they are immediately productive at your specific job. Contrast with Gen 1 specialists who only knew what you explicitly taught them.

Career

The AI Engineer Role

The AI Engineer bridges software engineering and AI/ML. They don't just train models — they build and deploy systems that use AI capabilities.

Software Engineering

Python, SQL, Git, REST APIs, CI/CD pipelines, Docker, cloud platforms (AWS/Azure/GCP)

LLM Integration

Prompt engineering, RAG (Retrieval-Augmented Generation), fine-tuning, LangChain, model evaluation

MLOps

Model versioning (MLflow), monitoring, drift detection, automated retraining, feature stores

Emerging

Model Context Protocol (MCP), agentic AI frameworks, multi-modal models, safety and alignment

Software Engineering vs ML Engineering

SE and ML engineering are siblings, not twins. The key differences change how you specify, test, debug, and maintain a system.

Professor's Notes from Webinar

These highlights are drawn directly from Prof. Shreyas Suresh Rao's spoken transcript of the Session 1 webinar. Items marked Exam Signal are things he explicitly flagged for the exam; Key Concept items are definitions and ideas he stressed or repeated; Professor's Comparison items are A vs B contrasts he drew.

⚑ Exam Signal

Exam format: "Midterm exam — 30%, closed book. Final exam — 40%, open book." The PPT itself will be sufficient for the final exam. Watermarked PDFs will be circulated before Session 13.

★ Key Concept

Functional vs Non-Functional Requirements: "There are two aspects: the functionality aspect and the non-functional aspect." Illustrated with Swiggy/Zomato — order processing is functional; scalability and performance are non-functional (quality attributes). The software architect is responsible for both.

★ Key Concept

Evolution of Software Development (flagged as "a very important slide"): Four stages — Waterfall (1980s–90s, rigid sequential phases) → Agile (early 2000s, fast iterations) → DevOps ("Agile and DevOps go hand in hand") → Cloud-Native (Agile + DevOps + Microservices). Know the sequence and what changed at each step.

⇄ Professor's Comparison

Monolithic vs Microservices: Monolithic = entire application as one module, single CI/CD pipeline, hard to change without breaking everything. Microservices = application divided into N independent services, each with its own CI/CD pipeline. "Once Netflix transitioned from monolithic to microservices, we saw a lot of traction." Most modern applications (Ola, Uber, Swiggy) now use microservices.

⇄ Professor's Comparison

Data Mining vs Machine Learning: "In data mining you want to mine/extract meaningful information. In machine learning, you want the machine to learn from labelled data to make predictions." Basic ML covers supervised learning (classification, regression); deep learning followed as data volumes grew too large for basic ML.

★ Key Concept

Data Journey (flagged as "a very important slide"): Timeline: 1960s files → 1970 IBM relational DB (E.F. Codd) → data mining → basic ML → cloud (AWS 2006) → big data/social media → deep learning → 2018 Transformer ("attention is all you need") → LLMs/foundation models. "Everything is driven by data — if so much data was not available, transformers would not have come into picture."

★ Key Concept

ADLC — Agentic Development Life Cycle: Introduced at session end. "ADLC stands for Agentic Development Life Cycle — it is about automation. AI is going to bring automation." Contrasted with SDLC. Includes AI-assisted generation of documents, design, and automation test scripts.

★ Key Concept

Observability in production: "Once I deploy a model into production, I cannot just keep quiet and relax. Observability becomes very important." ML systems need continuous monitoring of VM capacity, containers, and model behaviour post-deployment — it is not a one-time activity.
