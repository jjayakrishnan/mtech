Study Guide: All 5 Questions Solved

27 marks total · Try each question before revealing the answer

Question 1 · Agree / Disagree 6 marks · 2 each

Question 1

Challenge the accuracy of the below statements (agree/disagree) with proper justification.

  1. Microservices based projects recommend mono-repo patterns and single pipeline.
  2. A model-centric view of ML system focuses on entire end-to-end system, including data, infrastructure, deployment, and user experience.
  3. Pipe-and-Filter is an Architectural Pattern in software engineering which can be applied to machine learning systems.

How to answer agree/disagree questions

State clearly: Agree or Disagree. Give one precise reason why the statement is correct or wrong. One mark for the verdict, one mark for the justification.

1a · Microservices + mono-repo + single pipeline

Disagree.

Microservices architecture discourages both a mono-repo and a single pipeline. Each microservice owns its own codebase, its own repository, and its own independent CI/CD pipeline. This is the whole point of microservices: separate deployability. A mono-repo with a single pipeline creates exactly the tight coupling that microservices are designed to eliminate. If one service's pipeline fails, it should not block every other service.

Correct picture

Microservices → poly-repo (or at minimum separate pipeline-per-service) → independent deploy cycles. Mono-repo + single pipeline is the monolithic pattern, not the microservices pattern.

1b · Model-centric view focuses on the full end-to-end system

Disagree.

A model-centric view is the narrow view. It focuses only on the ML model itself — the algorithm, the training data, the hyperparameters, and the metrics. It treats everything else (data pipelines, infrastructure, deployment, monitoring, user experience) as someone else's problem.

The statement describes the system-centric (or data-centric / production-centric) view, which zooms out to the full socio-technical system. This is the view advocated in modern ML engineering, where the model is typically a small island in a much larger system.

Key distinction

Model-centric = the ML model is the centre of attention. System-centric = the whole pipeline, including data, infra, serving, and people, is the unit of concern.

1c · Pipe-and-Filter is an Architectural Pattern applicable to ML

Agree.

Pipe-and-Filter is an established Architectural Pattern in software engineering (defined in the POSA pattern catalogue). The pattern organises a system as a chain of independent processing units (filters) connected by typed data channels (pipes). Each filter transforms its input and passes output to the next.

ML systems are a natural fit. A typical ML preprocessing pipeline is: Raw Data → Clean → Normalise → Feature Engineer → Encode → Train/Predict. Each step is a stateless filter; the data flows through pipes. Scikit-learn's Pipeline class is a direct implementation of this pattern.

Why it works for ML

Each filter is independently testable and replaceable. You can swap a scaler or add a new feature step without touching the rest of the pipeline. This is exactly the modularity ML preprocessing needs.

Question 2 · GR4ML Analytics Design View 6 marks

Question 2 — Scenario

A bank collects customer information: income, credit score, employment status, and previous repayment history. The bank wants to determine whether a loan applicant is likely to default on a loan (Yes/No).

a. Draw the GR4ML Analytics Design View. 3 marks

b. Identify the Analytics Goal, two relevant Soft Goals, and two suitable algorithms. 3 marks

GR4ML Notation Reminder

The Analytics Design View shows what kind of analytical problem you are solving and how you will solve it. The four shapes you need:

  • Analytics Goal — the primary question (rounded rectangle / cloud shape) — "Predict loan default?"
  • Algorithm — the method chosen (hexagon) — e.g. Logistic Regression
  • Softgoal — a quality criterion without a hard threshold (cloud) — e.g. "High Accuracy"
  • Indicator — the measurable metric (rectangle with coloured bars) — e.g. AUC-ROC

Key arrows: Performs (solid arrow, Algorithm → Analytics Goal), Evaluates (double bar, Indicator → Softgoal), Influence (dashed arrow with triangle, Algorithm → Softgoal), Generates (dashed arrow, Analytics Goal → output).

2a · GR4ML Analytics Design View Diagram

Reading the diagram

  • Analytics Goal (rounded rect) = "Predict Loan Default (Yes/No)?" — this is a PredictionGoal because we are forecasting a future outcome.
  • Algorithms (hexagons) = Logistic Regression and Random Forest both Perform this goal (solid arrows).
  • Softgoals (clouds) = High Accuracy and Explainability — desirable qualities with no hard threshold.
  • Each algorithm Influences (dashed arrow) the softgoals differently — Logistic Regression is more explainable; Random Forest is typically more accurate.
  • Indicators (bar rectangles) = AUC-ROC and Recall — concrete metrics that Evaluate the softgoals.

2b · Analytics Goal, Soft Goals, Algorithms

ComponentAnswerJustification
Analytics GoalPredictionGoal: "Will this applicant default on the loan?" (binary Yes/No)We are forecasting a future event from historical features — this is a supervised classification / prediction task.
Softgoal 1High Accuracy / High RecallMissing a defaulter (false negative) costs the bank money. Recall on the "default" class must be maximised. No single threshold — it is a quality we want as high as possible.
Softgoal 2Explainability / Regulatory ComplianceBanking regulators (e.g. RBI, Basel) require that loan decisions be explainable to customers. A black-box model violates this. Explainability is a quality goal, not a metric.

Question 3 · Monolith vs Microservices ML Patterns 4 marks

Question 3

Differentiate between the Monolithic ML Pattern and the Microservices ML Pattern. Discuss suitable scenarios or use cases where each pattern would be preferred.

DimensionMonolithic ML PatternMicroservices ML Pattern
StructureAll ML components (data prep, training, serving, monitoring) reside in one deployable unitEach component is an independent service with its own API, data store, and deployment pipeline
DeploymentDeploy everything together — one build, one releaseDeploy each service independently — a new model version does not require redeploying the data pipeline
ScalingScale the whole application even if only inference is under loadScale only the bottleneck service (e.g. scale inference service ×10 while training runs on one node)
Team structureSmall team shares one codebase; low coordination overheadMultiple teams own separate services; each team can choose its own tech stack
Fault isolationOne bug can bring down the whole systemA failed training service does not affect the inference service
ComplexitySimple to build, test, and debug early onComplex distributed system; requires service discovery, API gateway, distributed tracing
LatencyIn-process calls are fast (no network hop)Inter-service network calls add latency; mitigated by async messaging (Kafka)

When to prefer Monolith

  • Early prototype or MVP — a startup building its first recommendation model. Speed to market matters more than scalability.
  • Small team (<5 engineers) — coordination cost of microservices outweighs the benefit.
  • Single ML model — a simple fraud detection model where all components share the same feature space.
  • Fast iteration — research phase where the model architecture changes weekly. One codebase is easier to refactor.

When to prefer Microservices

  • Multiple independent ML models — e.g. Google's production system with separate models for search ranking, ads CTR, translation, and spam detection. Each has different resource needs (GPUs vs CPUs).
  • Large teams — each team owns a service and deploys independently. No waiting for others.
  • Regulatory isolation — a bank that must audit its fraud model independently of its credit-scoring model.
  • Elastic scaling — an e-commerce platform that needs to scale its recommendation inference to 100× during a sale event without scaling training infrastructure.

The core trade-off

Monolith = simple now, painful later. Microservices = complex now, flexible later. The right choice depends on team size, model count, and whether independent deployability justifies the distributed-system overhead.

Question 4 · SAGA Pattern 6 marks

Question 4 — Scenario

An AI platform auto-trains and deploys ML models. Services: Data Validation Service, Model Training Service, Model Deployment Service, all coordinated by an AI Workflow Service.

Steps:

  1. Data scientist submits a request to train and deploy a predictive model.
  2. AI Workflow Service creates a workflow record and sends "validate dataset" → Data Validation Service.
  3. If valid, AI Workflow Service sends "train model" → Model Training Service.
  4. Model Training Service sends "model trained" → AI Workflow Service.
  5. AI Workflow Service sends "deploy model" → Model Deployment Service.
  6. Model Deployment Service deploys and sends "deployment successful". AI Workflow Service updates status to "completed" and notifies the data scientist.

a. Write all compensating transactions for the saga. 3 marks

b. Is this Orchestration or Choreography SAGA? Compare pros/cons of both. 3 marks

4a · Compensating Transactions

What is a compensating transaction?

A SAGA is a sequence of local transactions. If step N fails, you cannot "roll back" across services (no distributed 2-phase commit). Instead you run a compensating transaction for each already-completed step — an undo action that reverses its effect.

Key principle: compensate in reverse order

If step T5 (deploy) fails, you run C5 first, then C4, C3, C2, C1 — in reverse order. You only compensate steps that already succeeded. You never compensate a step that hasn't run yet.

4b · Orchestration vs Choreography SAGA

Verdict: this is Orchestration-based SAGA

The AI Workflow Service acts as a central coordinator. It explicitly tells each service what to do and when — "validate dataset", "train model", "deploy model". Services do not decide when to act; they respond to commands from the orchestrator. This is the defining characteristic of orchestration.

AspectOrchestrationChoreography
ControlCentral orchestrator service sends commands to each participantNo central coordinator — each service reacts to events published by others
CommunicationDirect command-response (synchronous REST or async commands via message queue)Event-driven (each service publishes events; subscribers react)
VisibilityEasy to trace — one service knows the full workflow stateHard to trace — no single place shows the overall status
CouplingParticipant services are loosely coupled to each other but tightly coupled to the orchestratorAll services are loosely coupled — they only share an event schema, not direct dependencies
Failure handlingOrchestrator detects failure and sends compensating commands in sequenceEach service publishes a "failed" event; upstream services must subscribe and compensate themselves
ProsSimple to reason about; single source of truth for workflow state; easy to add a new stepHighly decoupled; each service is independently deployable; scales well; no single point of failure
ConsOrchestrator is a single point of failure; can become a "god service" that knows too much; harder to scale independentlyHard to debug ("where is the workflow?"); tracing requires distributed tracing tools (Jaeger, Zipkin); complex to understand the whole flow
Best forWorkflows with a clear sequence and business rules (bank transaction, ML pipeline)Event-driven systems with many producers/consumers (IoT streams, e-commerce order events)

Question 5 · RAG & Batch vs Real-Time Serving 8 marks

Question 5a — 6 marks

An organisation maintains hundreds of PDFs (technical docs, company policies, SOPs). Employees ask questions in natural language and should receive accurate answers from these documents. As an AI Engineer, design a RAG-based solution. Using a suitable architecture diagram, explain how PDFs are processed, stored, retrieved, and used by an LLM to generate responses.

What is RAG?

Retrieval-Augmented Generation (RAG) = give an LLM only the relevant excerpts from your documents at query time, so it answers from facts, not from hallucinated training memory. Two phases: offline indexing (process and store all documents) and online querying (retrieve relevant chunks and ask the LLM to answer).

Phase 1 — Offline Indexing Pipeline

Phase 2 — Online Query Pipeline (real-time)

Step-by-step explanation

  1. PDF ingestion — All PDFs are loaded. A library like PyMuPDF or pdfminer extracts raw text page by page. Scanned PDFs pass through OCR (Tesseract).
  2. Chunking — Long documents are split into overlapping chunks (e.g. 512 tokens, 50-token overlap). Overlap ensures that no sentence is split across chunks and context is preserved at boundaries.
  3. Embedding — Each chunk is passed through a sentence embedding model (e.g. text-embedding-3-small from OpenAI, or an open-source all-MiniLM-L6-v2). This produces a dense vector of ~1536 floats that captures semantic meaning.
  4. Vector store — Vectors are stored in a vector database (Pinecone, Weaviate, ChromaDB, or pgvector). Each vector is linked to its source chunk text and metadata (filename, page, section).
  5. Query embedding — At query time, the employee's question is embedded using the same embedding model. This produces a query vector.
  6. Similarity search — The vector store performs approximate nearest-neighbour (ANN) search (cosine similarity) between the query vector and all stored chunk vectors. Top-K (e.g. K=5) most similar chunks are retrieved.
  7. Prompt construction — The retrieved chunks are inserted into a prompt template: "Answer this question using only the context below. If the answer is not in the context, say so.\n\nContext: [chunks]\n\nQuestion: [employee question]"
  8. LLM generation — The LLM (Claude, GPT-4, Llama 3) generates an answer grounded in the retrieved context. It cannot invent facts that are not in the context — this prevents hallucination.
  9. Citation — The answer is returned with source references (filename, page number) so employees can verify the original document.

Why RAG beats fine-tuning here

Fine-tuning bakes document knowledge into model weights — you must retrain whenever a policy changes. RAG keeps knowledge in the vector store — update a document, re-embed it, and the system instantly knows the new content. Much cheaper and faster for frequently changing enterprise documents.

5b · Batch vs Real-Time Serving 2 marks

Question 5b

Differentiate between batch serving and real-time serving patterns in machine learning systems.

DimensionBatch ServingReal-Time Serving
When predictions runScheduled job — run once per hour / day / week over a large datasetOn-demand — prediction runs the instant a request arrives (milliseconds)
LatencyHigh (minutes to hours acceptable)Low (<100ms typically required)
ThroughputVery high — millions of records per runLower — one (or a few) records per request
InfrastructureSpark / Airflow / BigQuery ML. No always-on model server needed.Always-on REST API (TorchServe, Triton, FastAPI + model). Requires auto-scaling.
CostLower — pay per batch run, no idle infrastructureHigher — server must be running 24/7 to avoid cold start latency
Feature freshnessFeatures can be stale (computed hours ago)Features must be fresh — often read from a real-time feature store
Use casesOvernight credit risk scores for all customers; weekly churn prediction emails; monthly fraud reportFraud detection at point-of-sale; real-time product recommendations; live content moderation
ExampleRun a BigQuery ML job at midnight to score all 10M customers and write results to a database for the morning dashboardCustomer adds item to cart → inference API called → recommendation returned in <50ms

Core trade-off

Batch serving = cheap and high-throughput but stale predictions. Real-time serving = fresh and low-latency but expensive and complex. Many production systems use both: batch for bulk scoring and real-time for user-facing interactions.

Common mistakes to avoid

Q1 — don't sit on the fence. State "Agree" or "Disagree" explicitly in your first line. Examiners mark the verdict separately from the justification. You get zero for a wishy-washy "it depends" without a clear stance.
Q2a — use the correct notation shapes. An Analytics Goal is NOT an Algorithm. Draw a hexagon for Algorithm and a rounded cloud shape for Analytics Goal and Softgoal. Label the arrows (Performs, Influence, Evaluates).
Q2b — justify your choices. Don't just name Logistic Regression. Say why it fits (binary output, interpretable, regulated environment).
Q3 — mention both sides. The question asks for scenarios where each is preferred. Half marks go to the differentiation table, half go to the use cases. Write at least two scenarios per pattern.
Q4a — compensate only completed steps. A common error is writing a compensating transaction for a step that hadn't run yet. The compensation is always for steps that already succeeded. Also note that compensating T1 (create workflow record) is always last — you can't notify failure before you know what failed.
Q4b — correctly identify the type first. The answer is clearly Orchestration because the AI Workflow Service issues commands. State this in the first sentence, then give the comparison table.
Q5a — the diagram is worth marks. Draw both phases: the offline indexing pipeline AND the online query pipeline. Name each component (embedding model, vector store, LLM). Explain each step in writing — the diagram alone is not enough for 6 marks.
Q5b — keep it structured. 2 marks = 1 mark per serving type, or half-mark per differentiating dimension. A table with 3–4 rows covers it safely in 5 minutes.

Golden rule for this exam

Every question rewards justification more than bare answers. A correct label with no reason scores partial marks. A correct label with a clear, specific reason scores full marks. Write one sentence of "why" for every technical claim you make.
