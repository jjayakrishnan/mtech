Session 5 · Overview

Four Patterns That Power Modern ML Systems

This session covers four architectural patterns you will encounter in every production ML system: CQRS (separate reads and writes), RAG (ground LLMs in real data), Monolith (single deployable unit), and Microservices (independent services).

Why these four? CQRS and RAG are the dominant patterns for ML system internals. Monolith vs Microservices is the dominant question for ML system boundaries. Together they cover the architecture of every major ML product you will build.
CQRS

Command Query Responsibility Segregation

CQRS separates operations that change state (commands) from operations that read state (queries). Each side is independently optimised.

Interactive: Traditional vs CQRS Architecture

Traditional: reads and writes compete for the same data store and locks.
Origin — Bertrand Meyer's CQS principle (Eiffel, 1980s): "Every method should either be a command that performs an action, OR a query that returns data, but NOT both." CQRS applies this at the architecture level.
Command side (write): trigger retraining | update features | log predictions
Query side (read): serve predictions | dashboards | metrics
Event stream (Kafka / Kinesis) propagates changes from write → read store
CQRS in ML

CQRS Applied to ML Systems

In ML, the command-query split maps naturally: model training and feature updates are commands; prediction serving and dashboards are queries.

Consistency

Eventual Consistency: The Trade-off

CQRS read stores are updated asynchronously — they lag behind the write store. This "eventual consistency" window is the price of separation.

Interactive: Consistency Timeline

Drag to change how long it takes all read stores to reflect a model update.
When eventual consistency is NOT acceptable: Real-time fraud detection. If a fraudster updates their account 2 seconds before a transaction and the read store hasn't caught up, the old prediction serves the fraud. Solution: synchronous write path for safety-critical queries.
RAG

Retrieval-Augmented Generation

RAG enhances an LLM by grounding its responses in external data. Instead of relying only on training-time knowledge, RAG fetches relevant documents first and injects them into the prompt.

The open-book analogy: An LLM without RAG is a student answering an exam from memory only. RAG gives the student the textbook: they look up relevant pages before writing. The answer is more accurate, more specific, and grounded in real data.

Why RAG?

LLMs have a training cutoff and don't know your company's proprietary data. RAG connects them to live, up-to-date, domain-specific knowledge without expensive retraining.

What RAG prevents

Hallucinations: the LLM can only cite what's in the retrieved context. It cannot invent facts that aren't there. "Answer only from the context provided."

Ingestion pipeline

Documents → Extract → Chunk → Embed → Index → Vector Store. Runs once and on updates. Uses Pipe-and-Filter pattern.

Query pipeline

User Query → Embed Query → Semantic Search → Retrieve Top-k → Build Prompt → LLM → Answer. CQRS Query side.

Monolith

Monolithic Architecture

A monolith packages all application functionality into a single deployable unit. It's the natural starting point for any new system.

FTGO example: Order Management, Restaurant Management, Delivery Management, Payment Processing — all in one WAR file, one MySQL database, one deployment.
PropertyAdvantageDisadvantage
DeploymentSingle WAR file; simple CI/CDFull redeploy for any change; high risk per deploy
CommunicationIn-process calls; fastest possibleAll modules share same failure domain
ScalabilityEasy to reason aboutCannot scale individual bottleneck components
TechnologyOne language, one frameworkTechnology lock-in; new tech means rewrite
TestingSimpler end-to-end testingBuild times grow; test suites slow down
ML Monolith pattern: Web app + model loading + preprocessing + prediction all in one Flask/FastAPI app. Perfect for prototypes. Problems emerge when: multiple models need different resource profiles, one model update requires full redeployment, a bug in preprocessing crashes prediction.
Microservices

Microservices Architecture

Microservices decompose an application into independent services, each owning its own database and API. The slider shows the scaling cost advantage.

Interactive: Scaling Cost Comparison

Move the slider to increase load. See how monolith vs microservices scale differently in cost.
Robert C. Martin's Single Responsibility Principle: "Gather together things that change for the same reason; separate things that change for different reasons." A Model Service changes when models retrain. A Logging Service changes when audit requirements change. They should be separate.

ML Microservices Demo

Gateway (port 8000) routes to Model Service (port 8001: loads fraud model, returns prediction) and Logging Service (port 8002: writes to PostgreSQL for drift monitoring). Each scales independently.

Instagram's Tech Stack

Python, React Native, JavaScript, Django, PostgreSQL, Objective-C, Nginx, Redis — different services use different tech. Microservices enable per-service technology choice.
