Session 4 · Overview

Quality Attributes and System Architecture

Two real events set the stakes. Flipkart's Big Billion Day 2014 crashed under load, losing crores of revenue. Hotstar served 25 million concurrent users during the Cricket World Cup with zero downtime. The difference was architecture — and architecture is driven by Quality Attributes.

The central question of this session: How do you specify what a system must be (not just what it must do), and how do you choose an architecture that delivers it?
Case Studies

Flipkart Fails, Hotstar Succeeds

Two Indian tech companies. Two dramatically different outcomes. The difference was identifying quality attributes before choosing architecture.

Interactive: Compare the Two Architectures

Flipkart: monolithic app, single database, no auto-scaling. QAs never formalised.
Root cause: Flipkart had not identified scalability and availability as key quality attributes before building. Hotstar explicitly specified: "Handle 25M concurrent streams at ≤2s buffer time" and built every infrastructure decision to satisfy that number.
Quality Attributes

What Is a Quality Attribute?

Quality Attributes (also called Non-Functional Requirements) specify how well a system does what it does — not what it does.

Formal definition (Len Bass): "A quality attribute is a measurable or testable property of a system that is used to indicate how well the system satisfies the needs of its stakeholders."
The qualification rule: Every functional requirement needs a QA annotation to be engineering-ready. "The system allows order placement" is incomplete. "The system allows order placement in ≤2 seconds at P95 under 1M concurrent users" is engineering-ready.

Functional Requirement

"The system allows users to place food delivery orders."

What: describes the feature.

+ Quality Attribute

"...in ≤3 seconds for 95% of requests at 10M orders/sec peak load, with 99.95% availability."

How well: makes it testable.
SMART QAs

Making QAs SMART

Every Quality Attribute must be SMART: Specific, Measurable, Attainable, Relevant, Time-sensitive. Vague QAs cannot be designed for, tested, or monitored.

QAVague VersionSMART Version
Performance"Fast response"Load restaurant menu in ≤2s; order placement ≤3s at P95
Availability"Always online"99.95% uptime annually; 24×7×365

Architecture

Quality Attributes Drive Architecture

The software architect's job is to select architectural patterns and technology choices that satisfy the specified quality attributes. QAs without an architecture are wishes. Architecture without QAs is guesswork.

Scalability → Microservices

Each service scales independently. The Order Service scales 5× at lunch peak; Delivery Service stays at 1×. Monolith would require 5× total system scale.

Availability → Circuit Breakers

If the ML inference service fails, a circuit breaker trips and falls back to a rule-based system. The overall application stays available even when the ML component is down.

Reproducibility → Model Registry

Every model version links to its exact training dataset version, code commit, and hyperparameters. Any version can be reproduced exactly.

Explainability → SHAP Layer

Every prediction request also generates a SHAP explanation showing which features drove the decision. Stored in audit log for regulatory inspection.
System Architecture

What Is System Architecture?

IEEE definition: "The software architecture of a program or computing system is the structure or structures of the system, which comprise software elements, the externally visible properties of those elements, and the relationships among them."

RAG Enterprise Chatbot Example: Hardware (AWS GPU instances) + Infrastructure (ECS, Docker, Kubernetes) + Software (API Gateway, LangChain orchestration) + ML (Embedding Model, Vector DB, LLM, Re-ranker) + Data (S3 policy documents, Pinecone vector store, PostgreSQL logs). All five layers must be specified in the architecture.

Hardware / Infrastructure

Cloud servers, GPU instances, storage, networking, containers, orchestration. The physical and virtual substrate.

Software Components

API gateway, authentication, business logic, orchestration layer. Traditional software components.

ML Components

Feature store, model registry, serving engine, monitoring. The AI-specific layer.

Data Layer

Vector database, relational DB, object storage, streaming pipelines. Where data lives and flows.
Design Patterns

Architectural Patterns

An architectural pattern is a general, reusable solution to a commonly occurring architecture problem. Expressed as {Context, Problem, Solution}.

Analogy: A house architect chooses from proven patterns (open-plan, split-level, courtyard) that have worked for similar families in similar climates. A software architect chooses from proven patterns (Pipe-and-Filter, CQRS, Event-Driven) that have worked in similar system contexts.

Pipe-and-Filter

The Pipe-and-Filter Pattern

Organises a system as a sequence of independent processing units (filters) connected by pipes. Each filter transforms data and passes it to the next. Step through the NLP pipeline below.

Interactive: NLP Sentiment Pipeline

Input: "I am not happy with the service!" — press Next to run through each filter.
Key benefit: Each filter can be swapped independently. Replace the tokeniser with a neural tokeniser, replace the vectoriser with BERT embeddings, replace the classifier with GPT — without changing any other filter.
Watch out: Pipe-and-Filter assumes stateless filters. If a filter must remember state across inputs (e.g., streaming models that update online), the pattern must be modified. Stateful filters break the independence guarantee.
Wrap-up

Session 4 Key Takeaways

QAs Before Architecture

Specify quality attributes before choosing any architectural pattern. The pattern exists to satisfy the QAs — not the other way around.

SMART QAs

Every QA must be Specific, Measurable, Attainable, Relevant, Time-sensitive. Vague QAs cannot be tested or designed for.

ML-Specific QAs

Accuracy, Explainability, Robustness, Fairness, Reproducibility, Data Drift Adaptability — these drive unique architectural choices not needed in traditional SE.

Pipe-and-Filter

The default pattern for ML preprocessing pipelines. Independent, swappable, testable filters connected by typed pipes. Modularity enables fast iteration.

Professor's Notes from Webinar

These highlights are drawn directly from Prof. Shreyas Suresh Rao's spoken transcript of the Session 4 webinar. Items marked Exam Signal are things he explicitly flagged; Key Concept items are stressed definitions; Professor's Comparison items are A vs B contrasts he drew.

⚑ Exam Signal

Assignment signal — "I will be asking": The professor explicitly stated: "Combining these three for a given use case — the functional requirement, the quality attribute, and the system architecture — this is what I will be asking." Expect scenario-based questions that combine all three.

★ Key Concept

Quality Attribute definition: "A quality attribute is a measurable or testable property that specifies how well a system meets the needs of the stakeholder along a specific dimension of interest. It is a qualification of the functional requirement. Without functional requirement, quality attributes have no significance."

⇄ Professor's Comparison

Availability vs Reliability: The professor used an analogy: "I am very fast in math. What is 10+2? 12. What is 10×2? 12." — fast but wrong. "Availability does not ensure reliability. A system may be available but need not be reliable. Reliability is about correctness."

⇄ Professor's Comparison

Scalability vs Performance: "Performance is mainly from the end-user perspective. Scalability is generally internal to the system — whether 100 people or 1 trillion people, the system should provide a response. Scalability does not imply reliability, and scalability need not talk about performance."

★ Key Concept

ML-only quality attributes (highlighted in red by professor): Accuracy, Robustness, Explainability, Fairness, Data Drift adaptability, Model Drift monitoring, Reproducibility. These are unique to ML systems and do not appear in traditional SE quality attributes.

⇄ Professor's Comparison

Data Drift vs Model Drift: "Data drift is the ability to adapt to changes in data over time. Model drift — you cannot keep using the same model over time; there could be degradation of model performance in the real-world scenario. Model drift is necessitated by data drift."

★ Key Concept

Fairness — Amazon resume screening case study: "The ML model must produce unbiased outcomes across groups. Amazon 2013 resume screening — it was only favouring the male demographic because most training data was male. This is an example of bias." Remember this as a canonical fairness failure example.

★ Key Concept

Flipkart Big Billion Day vs Hotstar ICC 2019 (prof said "just remember this"): Flipkart 2014 Big Billion Day — failure, monolith couldn't scale. Hotstar ICC 2019 World Cup — success with 25.3 million concurrent users, 3000 machines, 8 AWS regions, tsunami testing. These are the canonical case studies for architecture decisions.

⇄ Professor's Comparison

Architecture vs Design: "Architecture tells how many floors, how many rooms — the macro structure. Design is what happens inside each room — the micro-level decisions." A pattern at architecture level is called an architectural pattern; at design level it is called a design pattern.
