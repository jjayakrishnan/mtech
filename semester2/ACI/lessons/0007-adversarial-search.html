Lesson 7: Adversarial Search — Minimax & Alpha-Beta

Prof's Key Shift: From Single to Multi-Agent

"So far in all the algorithms we learned, how many agents were there? Only one — searching for the path and the goal. Now we ask: what if the number of agents is more? What if the environment is not fully observable?"

The Tic-Tac-Toe Analogy

"The girl says: by playing tic-tac-toe, I'm learning about chess. Why? Because she knows: if I win or not, I will make sure YOU are not winning. In chess too, we start with the ambition to win, but if you realize you can't win, you start playing defensively — ensure the other person also doesn't win. That's a draw."

The key cognitive shift: An agent cannot plan by only asking "what action gets me closer to my goal?" It must also ask: what will my opponent do to reduce my chance of success?

Zero-Sum Games

Prof's Definition

"In a zero-sum game, one player's gain is the other player's loss. The players have exactly opposite goals. If MAX gets +1, MIN effectively gets −1."

Real-life analogy: "In life, there are many zero-sum situations — a job interview, a promotion. One person's gain is another person's loss."

MAX and MIN — Critical Distinction

"MIN is NOT trying to directly win. He is trying to MINIMIZE the MAX score. Indirectly, by doing that, he is maximizing his own chance of winning."

PlayerGoalAt each node: picks
MAXMaximize utilityThe LARGEST value among successors
MINMinimize MAX's scoreThe SMALLEST value among successors

Every game that can be solved with adversarial search is described by six components:

Prof's Perspective on Utility

"Utility is always relative to a player's perspective. If O won: from MAX's view = −1, from MIN's view = +1. If X won: from MAX's view = +1, from MIN's view = −1."

Game Theory Basics

In adversarial search, we have two opponents making alternate moves. Each player is rational and tries to do the best they can, assuming the other player is also playing optimally.

Key Concepts

  • Zero-sum game: one player's gain is the other's loss (utilities sum to zero)
  • MAX player: wants to maximise the utility (our agent)
  • MIN player: wants to minimise the utility (the opponent)
  • Perfect information: both players see the full game state (chess, not poker)
  • Deterministic: no randomness in state transitions (chess, not backgammon)

Formal Definition

  • \( S_0 \): initial state
  • Player(\( s \)): whose turn it is at state \( s \)
  • Actions(\( s \)): legal moves at state \( s \)
  • Result(\( s, a \)): transition model — state after action \( a \)
  • Terminal-Test(\( s \)): is the game over?
  • Utility(\( s, p \)): final value for player \( p \) at terminal state \( s \)

Game Trees

Prof's Explanation

"A game tree represents all possible sequences of moves. At each level, players alternate. For tic-tac-toe, from the initial state, each player has multiple moves — draw the tree all the way to terminal states."

Expert players analogy: "Expert chess players like Pragyananda can visualize 2–3 levels for all combinations in their mind. Deep Blue can visualize even more. That's why they're champions — always proactive about 'if I make this move, what will he do?'"

Initialization Rule

"For all MAX nodes, initialize to \( -\infty \). For all MIN nodes, initialize to \( +\infty \). MAX nodes want to rise from \( -\infty \) toward positive. MIN nodes want to fall from \( +\infty \) toward negative."

A game tree unfolds all possible move sequences. Nodes are game states, edges are moves, and leaves are terminal states with a utility value.

The Core Dilemma

MAX cannot simply pick the path to the highest leaf (9) because MIN gets to choose at the intermediate level. If MAX picks Move B, MIN will choose Move E (utility 2), not Move F (utility 9). So MAX should pick Move A, where MIN picks the lesser evil: min(3,5) = 3 < what? Actually MAX gets guaranteed 3 from A, but from B only gets 2. So MAX picks A (guaranteed value 3).

Prof's Most Important Line

"MAX does not simply choose the branch with the BEST possible outcome. MAX chooses the branch whose WORST-CASE outcome is BEST."

"Minimax never promises the best result. It promises: you will not end up in the worst result. If the best possible terminal value is +15 and the worst is −10, minimax may choose a path leading to +8 — not the best, but you're guaranteed not to hit −10."

The Minimax Algorithm

Minimax computes the optimal strategy by working backwards from the leaves:

  • At MAX (α) nodes: pick the maximum of children's values
  • At MIN (β) nodes: pick the minimum of children's values
MINIMAX ALGORITHM (Recursive)
function MINIMAX-VALUE(state): if TERMINAL-TEST(state): return UTILITY(state) if PLAYER(state) == MAX: return max( MINIMAX-VALUE(child) for child in SUCCESSORS(state) ) if PLAYER(state) == MIN: return min( MINIMAX-VALUE(child) for child in SUCCESSORS(state) ) --- function MINIMAX-DECISION(state): return argmax( MINIMAX-VALUE(Result(state, a)) for a in Actions(state) )

Properties

PropertyValue
Complete?Yes (if tree is finite)
Optimal?Yes (against optimal opponent)
Time complexity\( O(b^m) \) — branching factor \( b \), depth \( m \)
Space complexity\( O(bm) \) (DFS)

Worked Example: 4-Level Minimax Tree

Problem Setup

4-level tree. Root is MAX. Each node has 2 children. Leaf pairs from left to right:

(8,7) | (3,15) | (9,8) | (-10,5) | (1,4) | (8,9) | (9,9) | (12,-3)

Step-by-Step Solution

Step 1: Level 3 (MIN nodes) — pick minimum of each pair
  • MIN of (8, 7) = 7
  • MIN of (3, 15) = 3
  • MIN of (9, 8) = 8
  • MIN of (-10, 5) = -10
  • MIN of (1, 4) = 1
  • MIN of (8, 9) = 8
  • MIN of (9, 9) = 9
  • MIN of (12, -3) = -3

Level 3 values: 7, 3, 8, -10, 1, 8, 9, -3

Step 2: Level 2 (MAX nodes) — pick maximum of each pair
  • MAX of (7, 3) = 7
  • MAX of (8, -10) = 8
  • MAX of (1, 8) = 8
  • MAX of (9, -3) = 9

Level 2 values: 7, 8, 8, 9

Step 3: Level 1 (MIN nodes) — pick minimum of each pair
  • MIN of (7, 8) = 7
  • MIN of (8, 9) = 8

Level 1 values: 7, 8

Step 4: Level 0 (ROOT — MAX) — pick maximum
  • MAX of (7, 8) = 8

Minimax value at root = 8

Optimal move: MAX chooses the right branch (leading to MIN value 8).

Prof's Simple 4-Level Example

This simpler tree directly illustrates the "best guaranteed outcome" insight:

Level 0 (MAX): Root Level 1 (MIN): Left subtree (L), Right subtree (R) Level 2 (MAX): L has children LL, LR; R has children RL, RR Level 3 (leaves): LL=[-1, 8] LR=[-3, -1] RL=[2, -3] RR=[4, 2]

Why Not the Best?

The best possible terminal value was 8 (LL branch). But minimax chose 2. Why? Because if Root chose the L branch (value −1), the MIN opponent would steer toward −1. By choosing R, Root guarantees at least +2 — not the best, but safe. This is the essence of minimax: best guaranteed outcome.

Static Evaluation Function (Preview)

Scope Note

This topic is introduced here briefly and covered in full detail in Lesson 8 (Alpha-Beta Pruning & MCTS).

When the game tree is too deep to evaluate fully, we stop at a certain depth and use a static evaluation function to estimate the value of a non-terminal state.

Chess Example (from the Professor)

Assign point values: Queen = 9, Bishop = 3, Pawn = 1. Static evaluation for white:

$$ \text{eval} = \sum(\text{white piece values}) - \sum(\text{black piece values}) $$

If white has 6 points and black has 13 points: \( 6 - 13 = -7 \). Black has the better position.

Alpha-Beta Pruning

Minimax explores the entire tree — but many branches are provably irrelevant. Alpha-Beta pruning eliminates these branches without affecting the result.

Key Definitions

  • \( \alpha \) = the best (highest) value MAX can guarantee so far along the path (lower bound on MAX's score)
  • \( \beta \) = the best (lowest) value MIN can guarantee so far along the path (upper bound on MIN's score)
  • Pruning condition: if at any node \( \alpha \geq \beta \), stop exploring that node's remaining children

Intuition

Think of it this way: if MAX already has a guaranteed option worth 7, and we discover a branch where MIN can force the value down to 5, MAX will never choose that branch. So we can stop exploring it — the exact value doesn't matter since MAX won't go there.

ALPHA-BETA PRUNING
function ALPHA-BETA(state, alpha, beta, player): if TERMINAL-TEST(state): return UTILITY(state) if player == MAX: value = -infinity for each child in SUCCESSORS(state): value = max(value, ALPHA-BETA(child, alpha, beta, MIN)) alpha = max(alpha, value) if alpha >= beta: // Beta cutoff break // Prune remaining children return value if player == MIN: value = +infinity for each child in SUCCESSORS(state): value = min(value, ALPHA-BETA(child, alpha, beta, MAX)) beta = min(beta, value) if alpha >= beta: // Alpha cutoff break // Prune remaining children return value --- Initial call: ALPHA-BETA(root, -infinity, +infinity, MAX)

When Does Pruning Happen?

  • At a MIN node (beta cutoff): MIN found a value \( \leq \alpha \) — MAX won't choose this path
  • At a MAX node (alpha cutoff): MAX found a value \( \geq \beta \) — MIN won't allow this path

Alpha-Beta Applied to Our Tree

Let's trace Alpha-Beta on the same tree. We process left-to-right (depth-first).

Full Alpha-Beta Trace (click to expand)

Start: ROOT (MAX), alpha=-inf, beta=+inf

Go left to Level 1 left MIN node, alpha=-inf, beta=+inf

  • Go to Level 2 left MAX: explore its children (Level 3 MIN nodes)
    • L3 node 1: MIN(8,7) = 7
    • L3 node 2: MIN(3,15) = 3
    Level 2 left MAX = max(7, 3) = 7. Update parent's beta? No — update alpha at MAX: alpha=7
  • Level 1 left MIN sees child value 7. Update beta = 7.
  • Go to Level 2 right MAX (under left MIN): alpha=-inf, beta=7
    • L3 node 3: MIN(9,8) = 8
    • At this MAX node: value = 8, alpha = max(-inf, 8) = 8
    • Check: alpha(8) >= beta(7)? YES! PRUNE!
    • L3 node 4: (-10, 5) PRUNED — never evaluated
    Level 2 right MAX returns 8 (but parent MIN already has beta=7, so it keeps 7)

Level 1 left MIN = min(7, 8) = 7. Root updates alpha = 7.

Go right to Level 1 right MIN node, alpha=7, beta=+inf

  • Go to Level 2 left MAX (under right MIN): alpha=7, beta=+inf
    • L3 node 5: MIN(1,4) = 1
    • L3 node 6: MIN(8,9) = 8
    Level 2 left MAX = max(1, 8) = 8
  • Level 1 right MIN sees child value 8. Update beta = 8. Check: alpha(7) >= beta(8)? No, continue.
  • Go to Level 2 right MAX (under right MIN): alpha=7, beta=8
    • L3 node 7: MIN(9,9) = 9
    • At this MAX node: value = 9, alpha = max(7, 9) = 9
    • Check: alpha(9) >= beta(8)? YES! PRUNE!
    • L3 node 8: (12, -3) PRUNED — never evaluated
    Level 2 right MAX returns 9 (but parent MIN has beta=8, keeps 8)

Level 1 right MIN = min(8, 9) = 8.

Root MAX = max(7, 8) = 8. Same answer as full Minimax!

Pruned Nodes

Two leaf groups were pruned and never evaluated:

  • L3 node 4: leaves (-10, 5) — pruned because MAX at Level 2 already found 8, exceeding MIN's beta of 7
  • L3 node 8: leaves (12, -3) — pruned because MAX at Level 2 already found 9, exceeding MIN's beta of 8

Result is identical to Minimax (value = 8), but we explored fewer nodes.

Alpha-Beta Complexity

CaseNodes ExploredEffective Branching Factor
Worst case (no pruning)\( O(b^m) \)\( b \)
Best case (perfect ordering)\( O(b^{m/2}) \)\( \sqrt{b} \)
Average (random ordering)\( O(b^{3m/4}) \)\( b^{3/4} \)

Best case means: alpha-beta with perfect move ordering can search twice as deep in the same time as plain Minimax! (Effective branching factor \( \sqrt{b} \) instead of \( b \).)

Monte Carlo Tree Search (MCTS)

When the game tree is too large for Minimax (e.g., Go with branching factor ~250), MCTS uses random simulations to estimate move quality.

Four Phases of MCTS

  1. Selection: From root, descend the tree using UCB1 to balance exploration vs exploitation
  2. Expansion: At a leaf node, add one or more children
  3. Simulation (Rollout): From the new node, play randomly to a terminal state
  4. Backpropagation: Update win/visit counts from the terminal node back to root
Selection Expansion Simulation Backpropagation ┌─[R]─┐ ┌─[R]─┐ ┌─[R]─┐ ┌─[R]─┐ │ │ │ │ │ │ │ +1 │ [A] [B] [A] [B] [A] [B] [A] [B] │ use UCB1 │ │ │ +1 [C] ←select [C]──[D]new [C]──[D] [C]──[D] │ │ +1 random Win! playout

UCB1 Formula

Upper Confidence Bound for Trees

$$ UCB1(s, a) = Q(s, a) + c \sqrt{\frac{\ln N(s)}{N(s, a)}} $$

  • \( Q(s, a) \) = average reward from taking action \( a \) in state \( s \) (exploitation)
  • \( N(s) \) = total visits to state \( s \)
  • \( N(s, a) \) = visits to child after action \( a \)
  • \( c \) = exploration constant (commonly \( \sqrt{2} \))
  • Second term grows when a child is under-visited (exploration)

When to Use MCTS vs Minimax

FactorMinimax + Alpha-BetaMCTS
Branching factorLow-moderate (chess ~35)High (Go ~250)
Evaluation functionNeeds a good heuristicNo heuristic needed (uses rollouts)
Anytime?Not naturallyYes — improves with more iterations
OptimalityOptimal (full search)Converges to optimal given enough time

Utility Functions (Exam Example)

In practice, we cannot search to terminal states in complex games. We use an evaluation function (heuristic) to estimate utility at non-terminal states.

Exam-Style Utility Function

$$ U = 3 \times (MAX_{wc} - MIN_{wc}) + 2 \times (MAX_{ap} - MIN_{ap}) $$

  • \( MAX_{wc} \): number of winning conditions for MAX
  • \( MIN_{wc} \): number of winning conditions for MIN
  • \( MAX_{ap} \): advantageous positions for MAX
  • \( MIN_{ap} \): advantageous positions for MIN

This is a linear weighted evaluation function. Positive U favours MAX; negative favours MIN.

Example Calculation

Suppose in a board game state: MAX has 4 winning conditions and 3 advantageous positions; MIN has 2 winning conditions and 5 advantageous positions.

$$ U = 3 \times (4 - 2) + 2 \times (3 - 5) = 3(2) + 2(-2) = 6 - 4 = 2 $$

Positive value → state slightly favours MAX.

Given this tree (left-to-right evaluation, MAX at root):

ROOT (MAX) / \ MIN MIN / \ / \ 5 3 8 ?

After evaluating the left subtree, MAX has alpha = 3 (since left MIN = min(5,3) = 3). Now we evaluate the right subtree. The first child of right MIN is 8, so MIN updates beta to 8. MIN's second child is unknown.

Minimax Solution:

  • Level 2 MAX nodes: max(6,4)=6, max(7,2)=7, max(5,9)=9, max(1,3)=3
  • Level 1 MIN nodes: min(6,7)=6, min(9,3)=3
  • Level 0 ROOT MAX: max(6,3)=6

Minimax value = 6. Optimal move: go left.

Alpha-Beta Trace:

Process left subtree first:

  • Left MIN → Left MAX: sees 6, then 4. MAX value = 6.
  • Left MIN → Right MAX: sees 7. alpha=7 at this MAX node. Then sees 2. MAX value = 7.
  • Left MIN = min(6, 7) = 6. Root alpha = 6.

Process right subtree (alpha=6, beta=+inf):

  • Right MIN → Left MAX: sees 5, then 9. MAX value = 9.
  • Right MIN updates beta = 9. Check: alpha(6) >= beta(9)? No.
  • Right MIN → Right MAX: sees 1. value=1. Then sees 3. value=3.
  • Right MIN = min(9, 3) = 3.

Root MAX = max(6, 3) = 6.

No pruning occurred in this particular tree with left-to-right ordering (the right subtree had the lower value, so we couldn't prune early). If we had evaluated right-to-left, we could have pruned!

Exam Tips

Watch Out For These

  • Know the 6 game components cold: initial state, player function, action function, result function, terminal test, utility function.
  • MAX and MIN always strictly alternate — if Root is MAX, level 1 is MIN, level 2 is MAX, and so on. This does NOT change mid-tree.
  • Minimax is used for two-player, zero-sum, perfect-information games only.
  • The minimax value at the root is the best guaranteed outcome for MAX assuming optimal play by both sides.
  • Prof's exam framing: "If I win or not, I will make sure you are not winning." — This captures why MIN plays to minimize MAX, not to maximize its own score directly.
  • When initializing: MAX nodes start at \( -\infty \) (want to go up); MIN nodes start at \( +\infty \) (want to go down).
